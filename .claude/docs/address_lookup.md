
## Address Lookup

### Overview

Address resolution happens in two stages: postcode-to-addresses (with UPRNs) and postcode-to-local-authority (to find the right scraper).

### Postcode to Addresses

The `/api/v1/addresses/{postcode}` endpoint calls the Mid Suffolk Placecube API to resolve a postcode into a list of addresses, each with a UPRN. This is done server-side in `api/services/address_lookup.py`.

### Postcode to Local Authority

Previously used the gov.uk API. Now uses a local parquet file (`api/data/postcode_lookup.parquet`) with about 1.6M postcodes mapped to LAD codes, queried via ibis/duckdb for fast local lookup. No external API calls needed.

The LAD code is then resolved to council metadata (name, homepage URL, scraper ID) via `api/data/lad_lookup.json`.

### Data Generation

The lookup data is generated by `scripts/lookup/create_lookup_table.py`, which:

1. Downloads the ONS ONSPD CSV (postcode directory)
2. Downloads UKBinCollectionData's `input.json` for council metadata
3. Builds `api/data/postcode_lookup.parquet` (postcode to LAD code)
4. Builds `api/data/lad_lookup.json` (LAD code to council name, URL, scraper ID)

### Admin Scraper Lookup

`api/data/admin_scraper_lookup.json` maps council website domains to scraper module IDs. Generated by `scripts/generate_admin_lookup.py` from scrapers' `URL` attributes. Used by the `/api/v1/council/{postcode}` endpoint to auto-detect the right scraper from a postcode.

### Client-Side Flow

The frontend (`api/static/address_lookup.js` and `api/static/app.js`) drives the user flow:

1. User enters postcode
2. Frontend calls `/api/v1/addresses/{postcode}` to get addresses with UPRNs
3. User selects address
4. Frontend calls `/api/v1/council/{postcode}` to find the council/scraper
5. Frontend calls `/api/v1/lookup/{uprn}?council=...` to get bin collection dates
